# Reporose Project Context

## Overview
Reporose is a repository scanner that builds a structural map of a codebase, including functions, imports, exports, and dependencies. It generates a `map.json` file that can be visualized in 3D or used for AI-assisted code navigation. Matter of Fact,,, A map for this repo can also be generated by running npm run analyze on the root of this project

## Architecture

### Core Components
- **Scanner** (`src/core/scanner.js`): Parses JavaScript/TypeScript files using Babel AST parser. Extracts functions, classes, imports, exports, and builds file metadata.
- **Mapper** (`src/core/mapper.js`): Computes dependency links between files, detects circular dependencies, and identifies dependency networks.

### AI Providers (`src/ai/`)
- **Ollama** (`ollama.js`): Local AI via Ollama API. Default model: `qwen2.5-coder:7b`
- **Ollama Cloud** (`ollama.js`): Cloud version of Ollama with API key authentication. Supports curated models like `gpt-oss:20b-cloud`
- **Anthropic** (`anthropic.js`): Anthropic Claude API via Messages API. Default model: `claude-haiku-4-5`
- **Groq** (via OpenAI): Fast inference via Groq API using OpenAI-compatible endpoint
- **OpenAI** (`openai.js`): OpenAI API compatibility
- **OpenRouter** (`openrouter.js`): Cloud AI with free tier
- **Local** (`local.js`): Local GGUF models via node-llama-cpp with tier-based presets (low/medium/high)
- **Heuristic** (`heuristic.js`): Rule-based, no AI - generates descriptions from AST patterns
- **None**: Disables AI description generation entirely
- **Summarizer** (`summarizer.js`): Orchestrates file/function description generation
- **Config** (`config.js`): Manages AI provider configuration and presets

### CLI & Server
- **CLI** (`src/cli/commands.js`, `bin/cli.js`): Commands: `init`, `analyze`, `map`, `summarize`, `serve`, `config`, `preset`
- **Server** (`src/server/server.js`): Express server serving 3D visualization

## Key Configuration
- Config stored at `<repo>/.reporose/config.json` or `~/.reporose/config.json`
- Cache stored at `<repo>/.reporose/cache/summaries.json`
- Output: `<repo>/.reporose/map.json`

## Default AI Settings

### Ollama Cloud
- Model: `gpt-oss:20b-cloud`
- Context: 32000 tokens
- Temperature: 0
- Max predict: 800 tokens
- API key required via `OLLAMA_API_KEY` environment variable

### Ollama (Local)
- Model: `qwen2.5-coder:7b`
- Context: 32000 tokens
- Temperature: 0
- Max predict: 10000 tokens
- Keep-alive: Model stays loaded during entire scan, unloaded only at end

### Anthropic
- Model: `claude-haiku-4-5`
- API key required via `ANTHROPIC_API_KEY` environment variable
- Streaming support via SSE

### Local Provider Tiers
- **Low**: ctx 2048, gpu_layers 0, file_tokens 180, fn_tokens 90 (1B-3B GGUF)
- **Medium**: ctx 4096, gpu_layers 24, file_tokens 280, fn_tokens 110 (7B GGUF)
- **High**: ctx 8192, gpu_layers 99, file_tokens 480, fn_tokens 140 (13B+ GGUF)

## File Processing Flow
1. **Phase 1 (Scan)**: Walk repo, parse AST, extract functions/imports/exports
2. **Phase 2 (Map)**: Compute dependency graph, detect cycles/networks
3. **Phase 3 (Summarize)**: AI generates file descriptions (function descriptions kept empty)
4. **Phase 4 (Visualize)**: 3D visualization via web interface

## Important Notes
- Function descriptions are intentionally left empty to reduce token usage
- Only file descriptions are generated by AI
- Model is kept "hot" (loaded) throughout the entire summarization phase
